# 校园众包平台公网基础安全版设计

## 1. 背景与目标

当前项目运行方式适合本机或小范围演示，不适合直接向不特定公网用户开放，主要原因如下：

- 后端当前缺少统一认证与鉴权机制，多个接口仅凭前端传入的用户名执行敏感操作。
- 用户密码以明文形式存储，不满足公网场景的基本安全要求。
- 余额充值接口对普通用户开放，存在被任意调用的风险。
- 前端存在“记住密码”“自动登录”等基于本地明文密码的单机化实现，不适合公网应用。

本次设计的目标是在尽量保留现有技术栈与产品结构的前提下，把项目改造成可面向不特定公网用户上线的“基础安全版”。

本次目标：

- 保留现有 `React + Vite + Spring Boot + MySQL` 架构，不做服务拆分。
- 允许普通用户自助注册并立即使用。
- 将认证改为 `JWT` 令牌模式。
- 将密码存储改为 `BCrypt` 哈希。
- 引入用户角色：`USER`、`ADMIN`。
- 新增管理员后台页面，用于查看用户和手动调整余额。
- 关闭普通用户公开充值入口。
- 保持数据库继续部署在用户个人电脑本机，不直接暴露到公网。
- 支持通过公网隧道服务快速上线。

本次不做的内容：

- 邮箱验证与找回密码。
- 短信验证、图形验证码、注册防刷。
- 完整审计日志系统。
- 多管理员权限分级。
- 云服务器、云数据库迁移。
- 支付系统接入。

## 2. 当前系统现状

项目结构：

- 前端：`campus-frontend`，`Vite + React`
- 后端：`campus-backend`，`Spring Boot + Spring Data JPA`
- 数据库：`MySQL`
- 部署方式：前端构建后打包进后端静态资源，由 Spring Boot 统一对外提供页面与 API

当前配置特征：

- 后端默认连接 `127.0.0.1:3306/campus_db`
- 前端开发时默认访问 `127.0.0.1:8080` 或 `127.0.0.1:8081`
- 后端接口普遍配置了 `@CrossOrigin(origins = "*")`
- 启动时可能自动打开浏览器
- 演示数据初始化默认可能开启

当前主要风险：

- `User.password` 为明文字段。
- 用户登录后没有服务端登录态，也没有统一权限判断。
- 资料修改、接单、完成任务、充值等操作依赖前端传参而不是当前登录身份。
- 普通用户可调用余额充值接口。
- 前端本地缓存了密码，不适合公网部署。

## 3. 总体架构设计

### 3.1 架构原则

本次改造遵循以下原则：

- 安全优先于最小改动，但避免不必要的大重构。
- 保持单体应用结构，降低部署复杂度。
- 认证、鉴权统一放在后端处理，前端不再承担身份可信性的判断。
- 普通用户和管理员共用同一套前端应用，通过角色控制页面和接口权限。

### 3.2 部署拓扑

部署后的拓扑如下：

- 用户个人电脑运行 `MySQL`
- 用户个人电脑运行 `Spring Boot` 应用
- Spring Boot 对外提供前端页面和 API
- 公网用户通过隧道服务访问应用公网地址
- 数据库仅允许本机连接，不开放公网端口

访问链路：

1. 公网用户访问隧道服务分配的公网地址
2. 隧道服务转发请求到本机应用端口
3. Spring Boot 返回前端页面或处理 API 请求
4. Spring Boot 通过本机回环地址连接 MySQL

### 3.3 运行边界

- 你的电脑必须持续开机、联网并运行应用，公网用户才能访问。
- 隧道地址可作为演示或早期上线入口，但稳定性和长期可用性弱于正式云服务器部署。
- 数据库留在本机是允许的，但只能由本机应用访问，不应直接暴露给公网。

## 4. 安全设计

### 4.1 认证机制

后端新增基于 `Spring Security + JWT` 的认证体系：

- 注册成功后创建普通用户账号，不自动发放管理员权限。
- 登录成功后返回 JWT 和当前用户信息。
- 前端后续请求统一在 `Authorization` 请求头中携带 `Bearer <token>`。
- 后端通过过滤器解析 JWT，恢复当前登录用户身份。
- 前端启动时通过 `GET /api/auth/me` 校验 token 并恢复登录态。

### 4.2 密码存储

密码存储统一改为 `BCrypt`：

- 数据库不再保存明文密码。
- 本次为了降低迁移复杂度，继续保留现有 `password` 字段名，但字段内容改为 `BCrypt` 哈希值。
- 注册时使用 `PasswordEncoder` 进行哈希。
- 登录时用 `BCrypt` 校验。

兼容旧数据的迁移策略：

- 启动时扫描现有用户密码字段。
- 若某条记录不是 `BCrypt` 格式，则按“旧明文密码”处理并自动转成哈希。
- 转换后保存回数据库，保证现有测试账号和历史账号可继续登录。

### 4.3 权限模型

角色只保留两种：

- `USER`：普通用户
- `ADMIN`：管理员

权限边界：

- `USER` 可注册、登录、查看任务、发布任务、接取任务、完成与自己相关的任务、查看并修改自己的资料、查看自己的余额与流水、收发自己有权限访问的消息。
- `ADMIN` 继承普通用户能力，并可访问管理员页面、查看用户列表、查看用户详情、手动调整用户余额。

### 4.4 管理员来源

管理员账号不通过公开接口创建，而是通过本地配置文件预置：

- 在本地配置中提供管理员用户名、管理员初始密码、管理员姓名。
- 应用启动时检查该管理员账号是否存在。
- 若不存在，则创建管理员账号。
- 若存在，则强制其角色为 `ADMIN`，并同步必要基础信息。
- 管理员密码同样使用 `BCrypt` 保存。

### 4.5 接口保护

接口访问策略：

- `/api/auth/register`、`/api/auth/login` 对匿名用户开放。
- 其余业务接口默认要求已登录。
- `/api/admin/**` 仅允许 `ADMIN` 访问。
- 所有身份敏感操作均以当前 token 对应用户为准，不再信任前端提交的用户名来决定操作对象。
- 生产环境前后端同源部署时不再使用 `@CrossOrigin(origins = "*")` 的宽松配置；如开发环境仍需跨域访问，应只允许明确的本地开发来源。

## 5. 数据模型设计

### 5.1 User

`User` 新增或调整如下字段语义：

- `username`：登录账号，保持唯一
- `password`：存储 `BCrypt` 哈希值
- `role`：`USER` 或 `ADMIN`
- `name`
- `email`
- `phone`
- `campus`
- `address`
- `bio`
- `balance`

约束与规则：

- 注册用户默认 `role=USER`
- 管理员初始化用户为 `role=ADMIN`
- 余额修改只能由管理员接口完成，普通用户接口不能直接修改余额

### 5.2 Task

`Task` 保持现有核心结构，但业务含义调整：

- `authorUsername` 始终由当前登录用户生成
- `author` 取当前登录用户展示名
- `assignee` 由当前登录用户接单时写入
- `completedAt` 在任务完成时由后端写入

权限约束：

- 任务创建者不能伪造发布人
- 普通用户不能替其他人接单
- 任务完成只能由发布者或管理员确认

### 5.3 BalanceRecord

保留余额流水表，用于记录管理员手动增减余额以及任务相关收支。

新增使用规则：

- 管理员手动调整余额时必须写入流水
- 流水记录需包含金额、调整后余额、类型、标题、说明、关联任务 ID（如无则为空）、创建时间

### 5.4 Message

消息结构可以暂不大改，但访问权限必须收紧：

- 只有任务发布者、接单者、管理员可查看该任务消息
- 只有任务参与者、管理员可发送消息

## 6. API 设计

### 6.1 认证接口

新增认证接口分组：

- `POST /api/auth/register`
- `POST /api/auth/login`
- `GET /api/auth/me`

接口行为：

- `register`：开放注册，成功后创建 `USER`
- `login`：返回 token 和当前用户摘要
- `me`：根据 token 返回当前登录用户资料

本次不做服务端 token 黑名单，退出登录由前端删除本地 token 即可。

### 6.2 普通用户接口

用户资料接口调整为只操作“当前用户”：

- `GET /api/users/profile`
- `PUT /api/users/profile`
- `GET /api/users/balance/me`

任务接口调整：

- `GET /api/tasks`：登录后可访问
- `POST /api/tasks`：根据当前登录用户创建任务
- `POST /api/tasks/{id}/accept`：当前登录用户接单
- `POST /api/tasks/{id}/complete`：发布者或管理员完成任务

消息接口权限调整：

- `GET /api/messages/{taskId}`
- `POST /api/messages`

以上接口均需验证当前用户是否有权访问目标任务会话。

### 6.3 关闭或移除的旧接口

以下旧行为不再保留给普通用户：

- 公开的 `POST /api/users/balance/recharge`
- 基于用户 ID 或前端传入用户名修改任意用户资料的接口行为
- 依赖前端直接指定作者、接单人、修改人身份的接口行为

如果为了兼容已有前端代码暂时保留旧路径，也必须在后端内部改为基于当前登录用户处理，不能继续信任请求体中的身份字段。

### 6.4 管理员接口

新增管理员接口：

- `GET /api/admin/users`
- `GET /api/admin/users/{id}`
- `POST /api/admin/users/{id}/balance-adjustments`

接口行为：

- 用户列表：返回基础信息，供后台搜索和选择
- 用户详情：返回单个用户资料、当前余额、最近余额流水
- 余额调整：管理员输入正数或负数金额，并附原因，后端更新余额并写入流水

余额调整规则：

- 调整金额必须是有效数字
- 允许正向增加或负向扣减
- 若扣减后余额小于 0，则默认禁止并返回错误
- 每次调整必须写明原因，便于后续追踪

## 7. 前端设计

### 7.1 登录态管理

前端从“本地保存密码并自动登录”的模型切换为“token 登录态”：

- 登录成功后保存 JWT
- 请求拦截器统一附带 token
- 页面刷新后通过 `GET /api/auth/me` 恢复当前用户
- token 无效时清空本地登录态并跳转到登录页

本地存储规则：

- 可保留“记住账号”，但只保存用户名
- 不再保存密码
- “自动登录”语义调整为：本地 token 仍有效时自动恢复登录

### 7.2 用户端页面调整

认证页：

- 登录和注册入口保留
- 注册仍要求基础资料
- 不再依赖本地密码进行自动登录

个人中心：

- 仅允许编辑当前登录用户资料

钱包页：

- 去掉普通用户充值表单
- 保留余额显示和流水记录展示

任务与消息页：

- 前端发请求时不再把身份字段作为可信来源
- 如果请求仍保留部分兼容字段，后端也必须忽略身份决定逻辑

### 7.3 管理员后台页面

在现有单页应用中增加 `AdminView`，不单独拆新项目。

管理员页面显示条件：

- 当前用户 `role=ADMIN` 时显示入口
- 普通用户不显示入口

第一版管理员页面功能仅包含：

- 用户列表
- 用户搜索
- 单个用户详情
- 余额调整表单
- 最近余额流水展示

交互要求：

- 明确区分“增加余额”和“扣减余额”
- 提交前展示确认提示
- 成功后刷新用户详情与流水
- 错误信息直接展示给管理员

## 8. 配置设计

新增本地配置项：

- 管理员用户名
- 管理员初始密码
- 管理员姓名
- JWT 密钥
- JWT 过期时间
- 是否自动打开浏览器
- 是否注入演示数据

推荐规则：

- 公网运行时关闭自动打开浏览器
- 公网运行时关闭演示数据初始化
- JWT 密钥必须由本地配置提供，不应硬编码到代码仓库

## 9. 部署设计

### 9.1 本机部署方式

继续保持如下部署方式：

- `MySQL` 安装并运行在你的电脑上
- Spring Boot 应用打包后在你的电脑上运行
- 前端随 Spring Boot 一起提供

### 9.2 公网暴露方式

使用隧道服务快速上线：

- 只暴露 Spring Boot 对外端口
- 不暴露 MySQL 端口
- 外部用户通过隧道地址访问应用

### 9.3 运行要求

- 电脑必须持续开机
- 应用进程必须持续运行
- 本地网络变化可能导致隧道地址变化或短时不可用
- 该方案适合早期上线和验证，不适合作为长期正式生产架构

## 10. 错误处理设计

统一错误语义：

- `400`：参数错误
- `401`：未登录、token 无效或过期
- `403`：已登录但无权限
- `404`：目标资源不存在
- `409`：业务冲突，例如用户名已存在、任务状态冲突、余额不足

前端处理规则：

- `401`：清除登录态并跳转登录页
- `403`：提示无权限
- 业务错误：直接展示后端消息

## 11. 测试设计

本次必须覆盖以下测试：

认证测试：

- 用户注册成功
- 重复用户名注册失败
- 登录成功
- 错误密码登录失败
- 旧明文密码迁移后登录成功

权限测试：

- 匿名用户无法访问受保护接口
- 普通用户无法访问管理员接口
- 管理员可访问管理员接口
- 普通用户不能修改其他用户资料

余额测试：

- 管理员增加余额成功
- 管理员扣减余额成功
- 扣减后小于 0 时失败
- 余额调整成功后流水正确写入
- 普通用户无法调用余额调整接口

任务测试：

- 当前登录用户发布任务时作者信息由后端写入
- 当前登录用户接单时接单人由后端写入
- 非发布者普通用户不能越权完成任务
- 发布者或管理员可以完成任务

消息测试：

- 非任务参与者无法查看消息
- 非任务参与者无法发送消息

## 12. 实施边界与风险

本设计能显著改善当前项目的公网上线安全性，但仍属于“基础安全版”，存在以下边界：

- 未实现注册防刷，仍可能遭遇恶意注册。
- 未实现邮箱找回密码，管理员需要人工处理账号问题。
- 未实现正式审计日志，管理员操作追踪能力有限。
- 应用运行在个人电脑上，稳定性、电源、网络、系统更新都会影响可用性。
- 隧道服务作为公网入口，稳定性和可控性有限。

因此，本次方案适合作为“可对不特定公网用户开放的基础版”，但后续若用户量增长或要长期运营，应迁移到云服务器并继续补充安全与运维能力。

## 13. 结论

本次改造以“在你的个人电脑上继续运行数据库和后端，但让不特定公网用户可访问”为约束，选择了改动量和安全性较平衡的方案：

- 保留现有技术栈和单体部署结构
- 用 `Spring Security + JWT + BCrypt + 角色权限` 补齐认证与鉴权
- 用本地配置预置管理员账号
- 用管理员后台替代普通用户公开充值
- 保持数据库只在本机运行
- 通过隧道服务快速提供公网访问入口

该方案完成后，项目将从“本机演示版”升级为“可公网使用的基础安全版”，并为后续迁移到云服务器保留清晰演进路径。
