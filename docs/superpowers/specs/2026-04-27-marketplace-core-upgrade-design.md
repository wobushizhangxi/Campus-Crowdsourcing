# 校园众包平台运营化第一阶段设计

Date: 2026-04-27
Scope: 将现有校园众包平台从“任务大厅演示版”升级为具备基础交易闭环、信用评价和校园认证标识的可运营雏形。

## 1. 背景

当前系统已经具备注册登录、任务发布、接单、任务完成、余额预扣与入账、任务消息、个人中心、钱包流水和管理员后台。核心问题是交易链路仍然过短：任务只有待接单、进行中、已完成三个状态，完成动作由发布者直接确认并立即打款，缺少接单人提交完成、发布者验收、取消、退款、纠纷处理、评价和认证标识。

对比同类任务或服务交易平台，第一阶段最需要补的是“交易可信度”，不是继续扩页面。可信度来自三件事：

- 任务状态可追踪，异常场景有出口。
- 交易完成后可评价，用户历史行为可沉淀。
- 用户是否经过校园认证有明确标识，降低陌生人交易风险。

## 2. 方案选择

### 2.1 可选方案

方案 A：只补任务生命周期。

- 优点：改动最小，能最快补上取消、提交、验收、退款。
- 缺点：用户选择接单人或发布者时仍缺少信用依据，平台可信度提升有限。

方案 B：任务生命周期 + 评价体系 + 校园认证标识。

- 优点：同时补齐交易闭环、用户信任和校园场景差异化；仍然可以沿用当前钱包模型，不引入真实支付复杂度。
- 缺点：涉及后端实体、接口、前端页面和管理端，实施量中等。

方案 C：一次性补全完整交易平台能力。

- 范围包括真实支付、提现、平台抽佣、实时 IM、举报仲裁、风控、搜索推荐、服务者主页。
- 优点：方向最完整。
- 缺点：一次实施风险过高，容易导致状态机、资金和权限规则同时失控。

### 2.2 推荐方案

采用方案 B。第一阶段只建设“运营化基础能力”，保留现有 React + Spring Boot + MySQL 架构，保留当前余额钱包，不接入真实支付渠道。后续支付、实时消息和推荐搜索可以在这个基础上迭代。

## 3. 目标

- 将任务流程扩展为可追踪状态机。
- 接单人需要提交完成，发布者需要验收后才结算。
- 支持任务取消、退款和管理员纠纷处理。
- 完成任务后双方可以互评。
- 用户资料展示评分、评价数量、完成数量和校园认证状态。
- 用户可提交校园认证申请，管理员可审核通过或驳回。
- 前端任务大厅、我的订单、个人中心、管理后台展示新增状态和操作。
- 现有登录、发布、接单、消息、钱包、管理权限继续可用。

## 4. 非目标

- 不接入微信、支付宝或银行卡支付。
- 不做提现和平台抽佣。
- 不把消息改成 WebSocket。
- 不做复杂推荐、地图定位或附近任务。
- 不做完整实名认证服务商接入。
- 不做独立移动端 App。

## 5. 任务生命周期设计

### 5.1 状态

新增明确的任务状态枚举语义，后端不再继续依赖自由字符串：

- `OPEN`：待接单。
- `ACCEPTED`：已接单，任务进行中。
- `SUBMITTED`：接单人已提交完成，等待发布者验收。
- `COMPLETED`：发布者验收通过，赏金结算给接单人。
- `CANCELLED`：任务已取消，按规则退款或不结算。
- `DISPUTED`：任务进入纠纷，等待管理员处理。

为了兼容已有数据，旧状态映射为：

- `open` -> `OPEN`
- `accepted` -> `ACCEPTED`
- `completed` -> `COMPLETED`

### 5.2 状态流转

正常流转：

1. 发布者发布任务，余额预扣，任务进入 `OPEN`。
2. 接单人接取任务，任务进入 `ACCEPTED`。
3. 接单人提交完成说明，任务进入 `SUBMITTED`。
4. 发布者验收通过，任务进入 `COMPLETED`，系统给接单人入账。

异常流转：

- `OPEN` 任务可由发布者取消，预扣金额退回发布者。
- `ACCEPTED` 任务不能直接取消；发布者或接单人如需终止任务，统一发起纠纷并进入 `DISPUTED`，由管理员决定退款或结算。
- `SUBMITTED` 任务可由发布者驳回，任务回到 `ACCEPTED`，需填写驳回原因。
- `SUBMITTED` 任务可由发布者发起纠纷，任务进入 `DISPUTED`。
- `DISPUTED` 任务只能由管理员处理为 `COMPLETED` 或 `CANCELLED`。

### 5.3 新增任务字段

在 `Task` 上补充第一阶段必要字段：

- `category`：任务分类，例如快递代取、跑腿、学习资料、技术帮助、其他。
- `campus`：校区。
- `location`：任务地点。
- `deadlineAt`：期望完成时间。
- `submittedAt`：接单人提交完成时间。
- `submissionNote`：完成说明。
- `cancelledAt`：取消时间。
- `cancelReason`：取消原因。
- `disputeReason`：纠纷原因。
- `resolvedAt`：管理员处理纠纷时间。
- `resolutionNote`：管理员处理说明。

第一阶段不做附件上传，完成凭证先用文字说明。

## 6. 钱包与结算设计

继续沿用当前余额模型：发布任务时从发布者余额扣减，完成后给接单人入账。

新增规则：

- 发布任务仍然预扣赏金，并写入流水。
- `OPEN` 任务取消时，将预扣赏金退回发布者，并写入退款流水。
- `SUBMITTED` 验收通过时，将赏金入账给接单人，并写入收入流水。
- `DISPUTED` 管理员判定退款时，将赏金退回发布者。
- `DISPUTED` 管理员判定完成时，将赏金入账给接单人。
- 任务一旦 `COMPLETED` 或 `CANCELLED`，不得重复结算。

为了降低第一阶段风险，不新增独立 escrow 表。后续接入真实支付时，再拆出冻结资金账户或交易单表。

## 7. 评价与信用设计

### 7.1 新增评价实体

新增 `TaskReview`：

- `id`
- `taskId`
- `reviewerUsername`
- `revieweeUsername`
- `reviewerRole`：`PUBLISHER` 或 `ASSIGNEE`
- `rating`：1 到 5
- `content`
- `createdAt`

约束：

- 只有 `COMPLETED` 任务可以评价。
- 发布者只能评价接单人一次。
- 接单人只能评价发布者一次。
- 管理员不能代写评价。
- 删除用户时保留评价记录，但用户名按现有删除策略匿名化。

### 7.2 用户信用摘要

用户摘要中增加：

- `averageRating`
- `reviewCount`
- `completedAsPublisherCount`
- `completedAsAssigneeCount`
- `verificationStatus`

第一阶段评分可以通过查询聚合生成，不强制在用户表冗余。若后续数据量上升，再做缓存字段。

## 8. 校园认证设计

### 8.1 用户认证状态

在 `User` 上新增：

- `verificationStatus`：`UNVERIFIED`、`PENDING`、`VERIFIED`、`REJECTED`
- `verificationCampus`
- `verificationStudentId`
- `verificationNote`
- `verificationSubmittedAt`
- `verificationReviewedAt`
- `verificationReviewer`

### 8.2 用户流程

用户在个人中心提交认证申请：

- 校区
- 学号
- 认证说明

第一阶段不上传图片，避免先引入文件存储。认证结果显示在个人中心、任务详情、订单详情和聊天资料区域。

### 8.3 管理员流程

管理员后台新增认证审核入口：

- 查看待审核申请。
- 通过认证，状态变为 `VERIFIED`。
- 驳回认证，状态变为 `REJECTED`，需填写原因。

### 9.1 任务接口

## 10. 前端设计

### 10.1 任务大厅

- 任务卡展示分类、校区、地点、截止时间、发布者评分和认证标识。
- 只展示 `OPEN` 任务。
- 接单前提示任务要求和发布者信用。

### 10.2 发布任务

发布表单新增：

- 分类
- 校区
- 地点
- 期望完成时间

发布前继续展示余额预扣说明。

### 10.3 我的订单

订单页面需要按角色区分动作：

- 发布者看到自己发布的任务，可以验收、驳回、取消、发起纠纷。
- 接单人看到自己接取的任务，可以提交完成、申请纠纷。
- 已完成任务展示评价入口。
- 已取消或已纠纷处理任务展示原因和处理结果。

### 10.4 个人中心

- 展示评分、评价数、完成任务数、校园认证状态。
- 增加校园认证申请表。
- 增加收到的评价列表入口。

### 10.5 管理后台

管理端新增两个能力区：

- 认证审核：查看、通过、驳回。
- 纠纷处理：查看 `DISPUTED` 任务，选择退款给发布者或结算给接单人，并填写处理说明。

## 11. 权限规则

- 匿名用户不能访问任何业务接口。
- 被封禁用户不能发布、接单、提交、评价或认证申请。
- 只有发布者可以取消 `OPEN` 任务。
- 只有接单人可以提交完成。
- 只有发布者可以验收或驳回提交。
- 只有任务发布者、接单人和管理员可查看任务消息。
- 只有管理员可处理纠纷和认证审核。
- 普通用户不能修改自己的认证结果。

## 12. 错误处理

统一使用现有响应结构：

- `400`：参数错误，例如评分不在 1 到 5。
- `401`：未登录或 token 失效。
- `403`：无权限或账号已封禁。
- `404`：任务、用户或认证申请不存在。
- `409`：状态冲突，例如已完成任务再次结算、重复评价。

前端对状态冲突要展示后端消息，并刷新任务和钱包状态，避免页面停留在旧状态。

## 13. 测试策略

后端新增集成测试：

- 发布任务后余额预扣。
- `OPEN -> ACCEPTED -> SUBMITTED -> COMPLETED` 正常流转。
- 验收通过后只结算一次。
- `OPEN` 取消后退款给发布者。
- 非任务参与者不能提交、验收、取消或评价。
- 已完成任务双方可各评价一次，不能重复评价。
- 用户提交认证后进入 `PENDING`。
- 管理员可通过或驳回认证。
- 管理员可处理纠纷为退款或结算。

前端验证：

- `npm run lint`
- `npm run build`
- 手动验证移动端和桌面端的任务大厅、发布任务、我的订单、个人中心、管理后台。

## 14. 实施分解

第一步：后端状态机和钱包结算。

- 引入任务状态枚举语义。
- 增加任务提交、验收、驳回、取消、纠纷处理接口。
- 补齐余额流水和重复结算保护。

第二步：评价体系。

- 新增评价实体、仓库、接口。
- 用户摘要返回评分和评价数。
- 前端增加评价入口和展示。

第三步：校园认证。

- 用户提交认证申请。
- 管理员审核认证。
- 前端展示认证标识。

第四步：页面整合和回归。

- 任务大厅、订单、个人中心、管理后台补齐新状态和操作。
- 统一错误提示和刷新逻辑。
- 跑后端测试、前端 lint/build。

## 15. 风险与缓解

- 风险：状态流转分散在控制器中会变复杂。
  - 缓解：实施时引入任务服务层，把状态检查和结算放到同一个事务边界中。
- 风险：预扣余额模式没有真实冻结账户，后续支付接入需要迁移。
  - 缓解：第一阶段明确不接真实支付；流水类型先命名清楚，为后续拆交易单留出口。
- 风险：评价聚合查询在用户量大时变慢。
  - 缓解：第一阶段用查询聚合，后续再做评分缓存。
- 风险：没有附件上传导致纠纷证据不足。
  - 缓解：第一阶段用文字说明完成闭环，文件证据作为后续独立功能。

## 16. 验收标准

1. 任务不能再从接单直接完成，必须先由接单人提交，再由发布者验收。
2. 取消、驳回、纠纷处理都有明确状态和原因记录。
3. 任务赏金不会重复结算，取消和纠纷处理会产生正确钱包流水。
4. 完成任务后双方可以互评，用户资料展示评分和评价数量。
5. 用户可以提交校园认证申请，管理员可以通过或驳回。
6. 任务大厅、订单页、个人中心、管理后台都能展示新增状态和关键操作。
7. 后端新增业务测试通过，前端 `lint` 和 `build` 通过。
