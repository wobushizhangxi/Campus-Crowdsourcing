# 离着远 Android 测试版 APK 设计

## 1. 背景与目标

当前项目已经具备可运行的前后端：
- 前端为 `Vite + React`
- 后端为 `Spring Boot`
- 前端 API 层已支持通过 `VITE_API_BASE_URL` 切换后端地址

当前用户目标已经从 Windows 安装包切换为 Android 测试版 APK，交付目标如下：
- 产出一个可在 Android 手机上手动安装的测试版 APK
- APK 内置前端页面，不依赖手机浏览器访问前端站点
- APK 通过公网后端地址访问现有 Spring Boot 服务
- 删除此前生成的 Windows 发布物，包括 `exe`、便携包、安装说明及相关中间产物

本次测试版 APK 固定使用以下公网后端地址：

`https://gather-break-long-que.trycloudflare.com`

该地址来自 `Cloudflare trycloudflare` 临时隧道，只用于测试，不视为正式生产域名。

## 2. 范围与非目标

本次范围内：
- 为现有前端接入 Android 容器方案
- 为 Android 构建准备固定测试环境变量
- 生成首个可侧载安装的测试 APK
- 清理旧的 Windows 发布物

本次不做：
- 不把 Spring Boot 后端打进手机本地运行
- 不做应用商店上架
- 不做正式签名发布包
- 不做推送、相机、定位等原生能力扩展
- 不解决临时隧道地址长期稳定性问题
- 不替换为正式云服务器或正式域名

## 3. 方案选择

已评估三条路线：
- `Capacitor` 本地壳 APK：内置前端静态资源，接口直连公网后端
- 远程 `WebView` APK：要求先有稳定可访问的前端站点
- `PWA/TWA`：更适合正式域名和正式发布

最终采用第一条：

`Capacitor + Android` 本地壳 APK

原因：
- 与现有 `Vite + React` 项目兼容性最好
- 对现有前端代码改动最小
- 可以直接复用现有路由、页面、登录状态管理和 API 封装
- 未来若要扩展原生能力，不需要推翻当前方案

## 4. 总体架构

首版 Android 测试版将采用以下结构：

1. 前端继续在 `campus-frontend` 内维护
2. 使用 `Vite` 构建前端静态资源
3. 使用 `Capacitor` 将构建结果同步到 Android 工程
4. Android 工程负责生成测试 APK
5. APK 内部页面通过 HTTPS 访问公网后端

数据流如下：

1. 用户在手机上打开 APK
2. APK 加载内置的前端静态页面
3. 前端通过 `axios` 请求 `https://gather-break-long-que.trycloudflare.com`
4. 后端返回登录、任务、消息等业务数据
5. 前端继续沿用现有 token 读写逻辑维持登录态

## 5. 前端改造设计

### 5.1 环境配置

前端构建需要新增 Android 测试版专用配置，核心要求：
- 构建 APK 时固定注入 `VITE_API_BASE_URL`
- 默认值即当前测试后端地址
- 该配置独立于本地开发环境，不影响现有 PC 开发流程

首版优先采用“构建脚本传参或专用环境文件”的方式，不把临时逻辑散落到业务代码中。

### 5.2 现有 API 层复用

`campus-frontend/src/services/api.js` 已支持从 `VITE_API_BASE_URL` 读取后端地址，因此 Android 首版不重写请求层，只做以下约束：
- APK 构建时必须提供非空 `VITE_API_BASE_URL`
- 生产型 Android 构建不再依赖 `DEV` 下的 `127.0.0.1` 回退地址
- 现有 JWT 请求头附加逻辑继续复用

### 5.3 UI 与交互

测试版 APK 不做新的页面设计，复用现有前端页面。首版只处理 Android 壳必要兼容项：
- 应用名称显示为 `离着远`
- 页面在移动端 WebView 中能正常打开和滚动
- 保持当前登录、任务列表、任务详情、消息等主要流程可用

## 6. Android 容器设计

### 6.1 容器方案

新增 `Capacitor` 相关依赖和 Android 工程，形成以下职责分层：
- `campus-frontend` 负责 Web 应用代码与构建产物
- `Capacitor` 负责 Web 资源到 Android 容器的桥接
- `android/` 工程负责 APK 编译与打包

### 6.2 应用标识

首版采用稳定的 Android 应用标识，默认值：

`com.lizheyuan.lzyapp`

显示名称使用：

`离着远`

该标识首版固定，不为测试版单独变更，以免后续正式化时重复迁移。

### 6.3 网络要求

Android 工程必须具备以下能力：
- 声明网络访问权限
- 允许通过 HTTPS 请求公网接口
- 不依赖本地局域网地址访问后端

由于后端地址为 HTTPS，首版不引入额外的明文流量放行配置。

## 7. 构建与产物设计

本次实现将补齐一条明确的 Android 构建链路，目标是让测试 APK 可以重复生成。建议形成如下流程：

1. 安装前端依赖
2. 使用测试公网地址构建前端
3. 同步到 `Capacitor Android`
4. 编译生成测试 APK
5. 输出到统一发布目录

预期产物：
- `release/离着远-android-test.apk`

如构建工具链限制导致默认输出路径不同，首版可以保留 Android 默认生成路径，再由脚本复制到 `release` 目录。

## 8. 旧 Windows 发布物清理设计

用户已明确要求删除此前生成的 Windows 发布物。本次实现要清理以下内容：
- `release/*.exe`
- `release/*portable.zip`
- `release/安装说明-*.md`
- `release/app-image`
- `release/jpackage-input`

清理策略：
- 只删除 Windows 发布相关文件，不删除运行日志与公网隧道日志
- 如果文件被进程占用，先停止相关程序，再执行删除
- 删除逻辑应限制在 `release` 目录内，避免误删其他工程文件

## 9. 错误处理与风险

### 9.1 临时公网地址风险

当前后端地址是 `trycloudflare` 临时地址，可能变化或失效。直接影响如下：
- 旧 APK 可能在地址失效后无法登录或加载数据
- 隧道关闭后，APK 虽能打开，但业务接口会失败

首版处理策略：
- 接受该限制，作为测试版已知约束
- 当公网地址变化时，重新生成测试 APK

### 9.2 后端不可用风险

若本地后端未启动，或隧道未连通，APK 将无法完成业务请求。首版不做复杂容灾，只要求：
- 保持前端现有网络错误提示可见
- 构建前确认后端地址可访问

### 9.3 Android 构建环境风险

若本机缺少 Android SDK、Gradle、或未完成 `Capacitor` Android 初始化，APK 无法产出。首版默认以当前机器为构建环境，必要时补脚本或说明文档。

## 10. 验证策略

实现完成后至少验证以下内容：
- 前端构建成功
- Android 工程同步成功
- APK 编译成功
- APK 安装到设备或模拟器后可以打开
- 登录接口可用
- 任务列表可加载
- 至少一条需要鉴权的接口请求成功
- 后端不可用时前端能给出错误反馈
- 旧 Windows 发布物已从 `release` 中移除

## 11. 实施边界

本次实现以“先出一个可安装、可联网、可登录的 Android 测试版 APK”为唯一目标，避免扩散范围。只要下面三点同时满足，就认为首版成功：
- 手机能安装 APK
- APK 能打开现有前端界面
- APK 能通过当前测试公网地址访问后端
